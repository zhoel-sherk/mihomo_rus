## С чего начать

- **Что за файл:** рабочая конфигурация Mihomo обычно называется `config.yaml` и лежит в **каталоге данных** ядра (параметр запуска `-d`; на Entware часто `/opt/etc/mihomo/`). На Keenetic фактический путь задаёт XKeen/обвязка — см. [`mihomo/info.txt`](mihomo/info.txt) в этом репозитории.
- **Как править:** сделайте резервную копию, отредактируйте YAML в редакторе с проверкой отступов, загрузите файл на роутер и **перезапустите** Mihomo. При синтаксической ошибке ядро выведет `Parse config error` и не поднимется — смотрите лог.
- **Дальше:** оглавление ниже; типы исходящих узлов подробно — [`docs/ru/proxies.md`](docs/ru/proxies.md); панели — [`docs/ru/web-ui.md`](docs/ru/web-ui.md).

Поднумерация подразделов (**### 1.**, **### 2.**, …) приведена к одному стилю там, где это уже было в тексте; в отдельных местах оставлены безномерные подзаголовки, если они логически не входят в одну цепочку.

## Быстрый старт (минимальный конфиг)

Готовый **публикуемый** пример без личных узлов — **[`mihomo/config.yaml`](mihomo/config.yaml)** (на устройстве его копируют как `config.yaml` в каталог `-d`). Ниже — укороченный скелет для ориентира (порты, `mode`, DNS, группа, `MATCH`):

```yaml
mixed-port: 10801
allow-lan: true
mode: rule
log-level: info
ipv6: false
find-process-mode: off

proxies: []          # сюда — ваш узел (см. docs/ru/proxies.md и § VLESS ниже)

proxy-groups:
  - name: PROXY
    type: select
    proxies:
      - DIRECT

rules:
  - MATCH,PROXY
```

Дополните `dns:`, `tproxy-port`, `rules` под свою схему (XKeen, nfqws) по разделам ниже.

## Глобальная конфигурация Mihomo (General Configuration)

Этот блок располагается в самом начале файла `config.yaml`. Он управляет поведением ядра Mihomo, сетевыми интерфейсами, системными лимитами, режимами логирования и настройками производительности.

---

### 1. Доступ из локальной сети (LAN Settings)

*   `allow-lan: true` — разрешает другим устройствам в локальной сети (ПК, смартфонам, ТВ) выходить в интернет через прокси-порты роутера.
*   `bind-address: "*"` — интерфейс для прослушивания входящих подключений. Символ `*` заставляет ядро слушать абсолютно все доступные IP-адреса роутера.
*   `lan-allowed-ips` — белый список подсетей, с которых разрешён доступ к LAN-прокси при `allow-lan: true`. По умолчанию в вики: `0.0.0.0/0` и `::/0` (весь IPv4 и IPv6).
*   `lan-disallowed-ips` — черный список подсетей или конкретных IP, которым запрещено сетевое проксирование (имеет приоритет над белым списком).
*   `authentication` — список пользователей и паролей для ручной авторизации на HTTP(S) и SOCKS прокси (в формате `"логин:пароль"`).
*   `skip-auth-prefixes` — список IP-адресов (например, `127.0.0.1/8`), которым разрешено подключаться к прокси без ввода пароля.

---

### 2. Режимы работы и системные параметры

*   `mode: rule` — основной рабочий режим. Трафик распределяется строго на основе правил из блока `rules:`. Другие варианты: `global` (весь трафик на один прокси) и `direct` (глобальное прямое подключение).
*   `log-level: warning` — уровень ведения логов. Для роутера рекомендуется `warning` или `error`, чтобы частая запись логов не забивала оперативную память и процессор. Вариант `silent` полностью выключает логи.
*   `ipv6: false` — включение или выключение обработки IPv6 трафика самим ядром. Настоятельно рекомендуется `false`, если вы используете `zapret/nfqws2` (подробнее — [совместная работа с nfqws2](#sec-nfqws)).

---

### 3. Энергосбережение и подбор процессов

*   `keep-alive-interval: 15` — интервал (в секундах) отправки TCP Keep-Alive пакетов для поддержания сессии.
*   `keep-alive-idle: 15` — максимальное время простоя перед отправкой Keep-Alive. Настройка этих двух параметров снижает энергопотребление подключенных мобильных устройств.
*   `disable-keep-alive: false` — запрет на полное отключение Keep-Alive пакетов.
*   `find-process-mode: off` — режим сопоставления трафика с процессом в ОС, где запущен Mihomo. Варианты: `always` (всегда), `strict` (**значение по умолчанию** в ядре), `off` (не матчить процессы). Для роутеров **рекомендуется `off`**: процессы клиентских ПК за NAT ядро не увидит, а лишние проверки грузят CPU.

---

### 4. Внешнее управление и Веб-интерфейс (REST API)

*   `external-controller: 127.0.0.1:9090` — IP и порт для управления ядром Mihomo через внешние панели (REST API). Для доступа со всей локальной сети адрес меняют на `0.0.0.0:9090`.
*   `secret: "ваш_пароль"` — секретный токен для авторизации внешних панелей в API вашего роутера.
*   `external-ui: /path/to/ui` — путь к папке со статическими файлами веб-панели управления (например, Yacd или Metacubexd), встроенной прямо в ядро.
*   `external-ui-url` — прямая ссылка на скачивание архива веб-панели с GitHub для автоматического обновления интерфейса.

Подробный гайд по панелям (**XKeen-UI**, **Yacd-meta**, **metacubexd**) и безопасному доступу к API: [**docs/ru/web-ui.md**](docs/ru/web-ui.md).

---

### 5. Оптимизация сетевого стека и кэширование

*   `unified-delay: true` — включает единый алгоритм расчета задержки (ping), убирая погрешность при рукопожатии (handshake) на разных типах прокси-протоколов.
*   `tcp-concurrent: true` — параллельное TCP-подключение. Ядро отправляет запросы сразу на все IP-адреса, выданные DNS, и берет в работу тот, который ответил самым первым. Значительно ускоряет открытие сайтов.
*   `interface-name: en0` — принудительное указание конкретного сетевого интерфейса роутера, через который Mihomo должен отправлять исходящий трафик в интернет.
*   `routing-mark: 6666` — задает цифровой маркер (fwmark) для всего исходящего трафика Mihomo в Linux-системах, что упрощает связку с цепочками `iptables/nftables`.

```yaml
profile:
  store-selected: true      # Запоминать выбранный вручную прокси в группах при перезагрузке роутера
  store-fake-ip: true       # Кэшировать таблицу fake-ip на флешке, чтобы домены сохраняли выданные IP
```

---

### 6. Управление базами GEO-данных (Критично для Keenetic)

*   `geodata-mode: true` — GeoIP читается из **`GeoIP.dat`** (protobuf, как у v2ray); списки доменов по-прежнему из **`GeoSite.dat`**. При **`false`** (частый случай на роутере) GeoIP обычно берётся из **`geoip.metadb`** / связанных MMDB-файлов, а не из `GeoIP.dat` (см. [`mihomo/README-GEO.md`](mihomo/README-GEO.md), симлинки из каталога Xray `dat` — [`docs/ru/geo-xray-dat.md`](docs/ru/geo-xray-dat.md)).
*   `geodata-loader: memconservative` — **самый важный параметр для домашних роутеров**. Режим `memconservative` оптимизирует загрузку тяжелых баз гео-данных в оперативную память. Он удерживает базы на флеш-накопителе, загружая в ОЗУ только нужные фрагменты, что предотвращает падение ядра по нехватке памяти (`OOM Panic`).
*   `geo-auto-update: false` — автоматическое обновление баз самим ядром. Если вы используете XKeen, эту функцию **нужно выключить (`false`)**, так как XKeen сам обновляет базы через свои скрипты по крону.
*   `geox-url` — ссылки на скачивание файлов GeoIP / GeoSite / **country.mmdb** / **GeoLite2-ASN.mmdb** (поля `geoip`, `geosite`, `mmdb`, `asn`); см. [вики — geox-url](https://wiki.metacubex.one/en/config/general/#custom-geo-download-address).
*   `global-ua: clash.meta` — кастомный User-Agent, который ядро отправляет серверам при скачивании обновлений и подписок (по умолчанию в ядре уже `clash.meta`).
*   `geo-update-interval: 24` — период автообновления Geo-файлов в **часах** (работает вместе с `geo-auto-update: true`).
*   `etag-support: true` — использовать заголовок ETag при скачивании внешних ресурсов (по умолчанию включено).
*   `external-controller-cors` — CORS для REST API (`allow-origins`, `allow-private-network`).
*   `external-controller-unix` / `external-controller-pipe` — API через Unix socket или именованный канал Windows (**доступ без проверки `secret`** — включайте только осознанно).
*   `external-controller-tls` — HTTPS для API; требует блока `tls:` с `certificate` и `private-key`.
*   `external-ui-name` — подпапка внутри `external-ui` для веб-панели при автообновлении через `external-ui-url`.
*   `tls:` (в корне) — сертификаты для HTTPS API; с v1.19.18 поддерживается автоперезагрузка при изменении файлов на диске.

## Глобальные порты прокси (Proxy Ports) в Mihomo

Эти параметры прописываются в корневом блоке конфигурации (вне секции `listeners:`). Они открывают стандартные точки входа для трафика. Их главное отличие от `listeners` — максимальная простота записи, когда не нужно городить многострочные списки.

---

### 1. HTTP(S) Порт прокси
*   `port: 7890` — открывает классический порт для HTTP/HTTPS трафика. Подходит для явного проксирования трафика браузеров или системных утилит, которые не умеют работать со спецификацией SOCKS5.

---

### 2. SOCKS Порт прокси
*   `socks-port: 7891` — открывает порт с полноценной поддержкой протоколов SOCKS4, SOCKS4a и SOCKS5. SOCKS5 является более продвинутым протоколом по сравнению с HTTP, так как умеет корректно пропускать через себя не только веб-страницы, но и сторонние сетевые потоки (например, трафик мессенджеров или торрент-клиентов).

---

### 3. Смешанный порт (Mixed Port)
*   `mixed-port: 7892` — универсальный «умный» порт, совмещающий в себе функции HTTP и SOCKS5 одновременно. 
*   **В чем удобство:** Ядро Mihomo автоматически распознает тип входящего запроса от подключаемой программы и обрабатывает его по нужному протоколу. Именно этот параметр чаще всего используется в связках с XKeen на роутерах Keenetic для ручного проксирования приложений.

---

### 4. Порты прозрачного проксирования (Transparent Proxy)
*Эти порты используются операционной системой роутера (Linux/KeeneticOS) для незаметного перехвата всего сетевого трафика ваших домашних устройств:*

*   `redir-port: 7893` — порт прозрачного проксирования через системный механизм `REDIRECT`. 
    *   **Ограничение:** Работает в Linux и macOS, но технически способен перехватывать и обрабатывать **исключительно TCP-трафик** (обычные сайты). Игры и стримы (UDP) через него не пойдут.
*   `tproxy-port: 7894` — порт для прозрачного проксирования через механизм `TPROXY`.
    *   **Преимущество:** Работает только в Linux (Android/KeeneticOS). Способен без ограничений перехватывать **одновременно и TCP, и UDP-трафик**. Это самый мощный инструмент для обеспечения работы онлайн-игр на консолях и тяжелого медиа-трафика напрямую через роутер.

---

### Пример лаконичного классического блока в начале конфига:

```yaml
# Классическая схема портов без использования многострочных listeners
mixed-port: 10801
redir-port: 1182
tproxy-port: 1181
allow-lan: true
```

## Виртуальный сетевой интерфейс TUN в Mihomo

Блок **tun** создает внутри операционной системы роутера или ПК виртуальную сетевую карту. Это позволяет Mihomo перехватывать абсолютно весь входящий и исходящий трафик на уровне сетевых драйверов ядра Linux, полностью заменяя капризные правила таблиц маршрутизации iptables/nftables. 

---

### 1. Базовые параметры активации

*   `enable: true` — глобальное включение режима виртуальной сетевой карты TUN.
*   `device: utun0` — ручное указание имени создаваемой виртуальной сетевой карты. В операционных системах macOS название обязано начинаться строго с префикса `utun` (например, `utun0`, `utun1`).
*   `mtu: 9000` — максимальный размер полезного блока данных (Maximum Transmission Unit), передаваемого в одном пакете. Изменение значения влияет на предельную скорость пропуска трафика процессором. Обычным пользователям рекомендуется оставлять значение по умолчанию.

---

### 2. Выбор сетевого стека (stack)
*Параметр определяет технологический движок, который будет обрабатывать сетевые пакеты внутри виртуальной карты. Доступно 3 варианта:*

*   `gvisor` — движок по умолчанию. Реализует сетевой протокол внутри пользовательского пространства (user-space), изолируя трафик. Обеспечивает максимальную безопасность, но тратит ресурсы CPU роутера на переключение контекста ядра.
*   `system` — использует стандартный сетевой стек самой операционной системы роутера (ядра Linux). Работает значительно быстрее, тратит меньше оперативной памяти и ресурсов процессора, но требует ручного проброса правил в брандмауэрах (Windows/Linux).
*   `mixed` — **наиболее рекомендуемый режим**. Смешанный стек, где весь TCP-трафик обрабатывается быстрым системным движком (`system`), а тяжелый и капризный UDP-трафик уходит под управление изолированного `gvisor`. Обеспечивает лучший баланс скорости и стабильности.

---

### 3. Автоматизация маршрутизации и перехвата

*   `auto-route: true` — автоматически настраивает глобальную таблицу маршрутизации ОС, принудительно заворачивая весь трафик устройства внутрь созданной карты TUN.

    **Keenetic и XKeen:** включение **`auto-route: true`** на самом роутере часто означает захват **дефолтного маршрута** всей системы и легко конфликтует с уже настроенной цепочкой **nftables → nfqws → tproxy Mihomo**. Если вы читаете только этот раздел TUN и «включаете как в гайде на ПК» — получите петли или обход nfqws. Сначала прочитайте [§ Совместная работа с nfqws2](#sec-nfqws); на типичном Keenetic+XKeen чаще используют **прозрачный tproxy**, а не полный TUN-захват.

*   `auto-redirect: true` — эксклюзивная функция для Linux. Автоматически перестраивает цепочки правил брандмауэров `iptables/nftables` для прозрачного перенаправления TCP-сессий. Требует обязательного включения параметра `auto-route`. На роутерах работает "из коробки" без ручного вмешательства.
*   `auto-detect-interface: true` — заставляет Mihomo автоматически определять ваш основной физический сетевой интерфейс, через который роутер подключен к провайдеру (出口网卡). Полезно для устройств с несколькими активными интернет-кабелями.
*   `dns-hijack` — жесткий перехват DNS. Список адресов и портов, трафик на которые Mihomo принудительно заберет в свой внутренний криптографический DNS-модуль. Конструкция `any:53` означает перехват абсолютно любого DNS-запроса в сети.
*   `strict-route: true` — режим строгого контроля утечек. Полностью блокирует любой трафик, идущий в обход созданной карты TUN, исключая незаметные утечки реального IP-адреса через альтернативные интерфейсы или системные службы DNS брандмауэра Windows.

---

### 4. Тонкие сетевые настройки Linux

*   `gso: true` — включение аппаратного или программного механизма Generic Segmentation Offload (только для Linux). Позволяет сетевой карте обрабатывать большие куски данных единым блоком, снижая нагрузку на центральный процессор роутера.
*   `gso-max-size: 65536` — устанавливает предельно допустимую длину дата-блока для механизма GSO.
*   `udp-timeout: 300` — время жизни (тайм-аут) для UDP-трансляций в секундах (по умолчанию 5 минут). По истечении этого времени неактивная UDP-сессия закрывается, освобождая оперативную память.
*   `iproute2-table-index: 2022` — цифровой индекс виртуальной таблицы маршрутизации, которую создает скрипт `auto-route` в подсистеме Linux `iproute2`.
*   `iproute2-rule-index: 9000` — стартовый цифровой индекс для правил маршрутизации в системе `iproute2`.
*   `endpoint-independent-nat: false` — включение независимого от конечной точки NAT. Слегка снижает общую производительность сети, поэтому включать его без прямой необходимости не рекомендуется.

---

### 5. Фильтрация IP-адресов и диапазонов (CIDR)

*   `route-address-set` — список именованных правил (Rule-Set) с диапазонами IP-адресов. Весь трафик, совпадающий с этим списком, будет принудительно направлен в TUN. Вся остальная сеть пойдет в обход. Работает только в Linux через `nftables`. Конфликтует с глобальным параметром `routing-mark`.
*   `route-exclude-address-set` — список правил-исключений. Трафик к IP-адресам из этого набора гарантированно пойдет напрямую мимо виртуальной карты TUN.
*   `route-address` — ручное указание диапазонов сетей, которые нужно завернуть в TUN вместо глобального роутинга по умолчанию (обычно используются маски `0.0.0.0/1` и `128.0.0.0/1`).
*   `route-exclude-address` — жесткий список подсетей-исключений из виртуальной карты. Сюда обязательно заносят локальные диапазоны (например, `192.168.0.0/16`), чтобы устройства не теряли доступ к домашним сетевым ресурсам и панели роутера.

---

### 6. Фильтрация интерфейсов и пользователей (Linux & Android)

*   `include-interface` / `exclude-interface` — белый и черный списки физических сетевых карт (например, `eth0`, `wlan0`), трафик которых разрешено или запрещено перехватывать модулю TUN. Запрещено использовать одновременно.
*   `include-uid` / `include-uid-range` — список цифровых идентификаторов пользователей (UID) в ОС Linux, чей трафик подлежит роутингу через TUN.
*   `exclude-uid` / `exclude-uid-range` — исключение конкретных пользователей операционной системы из обработки виртуальной картой.
*   `include-android-user` — эксклюзив для Android-систем. Позволяет изолировать работу прокси на конкретных профилях смартфона (например, `0` — основной владелец, `10` — второе пространство / системный клон).
*   `include-package` / `exclude-package` — белый и черный списки системных пакетов приложений Android (App Package Name). Позволяет принудительно запустить проксирование только для Google Chrome (`com.android.chrome`) или исключить системные утилиты авторизации (`com.android.captiveportallogin`).

---

### 7. Устаревший синтаксис (Депрекация)
*Данные параметры признаны устаревшими и будут полностью удалены в будущих релизах ядра Mihomo. Использовать их в новых конфигах запрещено:*
*   `inet4-route-address` / `inet6-route-address` — заменены единым универсальным параметром `route-address`.
*   `inet4-route-exclude-address` / `inet6-route-exclude-address` — заменены единым параметром `route-exclude-address`.

---

### Шаблон эталонной конфигурации блока `tun`

```yaml
tun:
  enable: true
  stack: mixed                      # Стабильный смешанный стек TCP-System / UDP-Gvisor
  device: utun0
  auto-route: true
  auto-redirect: true               # Автоматическая настройка nftables в Linux
  auto-detect-interface: true       # Автопоиск интернет-кабеля провайдера
  dns-hijack:
    - any:53                        # Перехват всех DNS-запросов
  strict-route: true                # Полная защита от утечек трафика
  mtu: 9000
  udp-timeout: 300
  route-exclude-address:
    - 192.168.0.0/16                # Исключаем домашнюю локальную сеть
    - fc00::/7                      # Исключаем локальный IPv6 диапазон
```

## DNS Hosts в Mihomo

Блок **hosts** — это встроенный аналог системного файла `/etc/hosts`. Он жестко связывает домен с конкретным IP-адресом. Mihomo отдает этот IP-адрес локальным устройствам мгновенно (за 0 мс), полностью игнорируя внешние DNS-сервера в интернете.

---

### Системные переключатели
*Эти параметры прописываются в корневом блоке `dns:` для активации логики:*

*   `use-hosts: true` — заставляет ядро читать внутренний блок настроек `hosts`.
*   `use-system-hosts: true` — разрешает Mihomo считывать записи из файла `hosts` самой операционной системы роутера Keenetic.

---

### Синтаксис масок Mihomo
*В отличие от старого Clash, Mihomo поддерживает продвинутые регулярные выражения:*

*   `'example.com': 1.1.1.1` — точное совпадение. Сработает только на этот конкретный домен.
*   `'*.example.com': 2.2.2.2` — wildcard-маска Clash. Покрывает поддомены (`www.example.com`, `mail.example.com`), но **не** сам корень `example.com`.
*   `'+.example.com': 3.3.3.3` — максимальный охват. Задает IP-адрес как для корня `example.com`, так и для абсолютно всех его поддоменов.

---

### Сценарии применения

#### Локальная сеть (LAN)
Назначение понятных доменных имен домашним серверам, сетевым хранилищам или панелям управления вместо запоминания их локальных IP-адресов.
```yaml
dns:
  hosts:
    'nas.lan': 192.168.1.10
    'home.assistant': 192.168.1.15
    'router.keenetic': 192.168.1.1
```

#### Блокировка рекламы и ботнетов (Null-Routing)
Отправка рекламных доменов, трекеров телеметрии или управляющих серверов вирусов на несуществующий адрес `0.0.0.0`. Запросы мгновенно обрываются на уровне процессора роутера, экономя оперативную память и сетевой трафик.
```yaml
dns:
  hosts:
    '*.telemetry.microsoft.com': 0.0.0.0
    '*.doubleclick.net': 0.0.0.0
    'hacksfestparadise.net': 0.0.0.0
```

#### Прямой обход блокировок (IP-Pinning)
Если провайдер заблокировал сайт методом подмены DNS-ответов (DNS-Hijacking), но сам IP-адрес сервера в интернете доступен напрямую, его можно прописать жестко. Сайт начнет открываться в режиме `DIRECT`, вообще не нагружая арендованный VPS-сервер.
```yaml
dns:
  hosts:
    'some-clean-vps-site.com': 185.199.108.153
```

## DNS: блок `dns:` (встроенный резолвер)

Встроенный DNS-модуль Mihomo задаётся корневым ключом `dns:`. Он отвечает за то, **какими серверами** и в каком режиме резолвятся домены для маршрутизации и для самого ядра. Подробно: [DNS в вики](https://wiki.metacubex.one/en/config/dns/).

### Включение и прослушивание

*   `enable: true` — использовать встроенный DNS; при `false` для резолва опираются на системный стек.
*   `listen: 0.0.0.0:1053` — приём DNS (UDP/TCP) на указанном адресе.
*   `ipv6: false` — при `false` для AAAA-записей возвращается «пустой» ответ (нет IPv6-адресов из DNS модуля).
*   `prefer-h3: false` — для DoH отдавать приоритет HTTP/3.
*   `cache-algorithm` — алгоритм кэша: `lru` (по умолчанию) или `arc`.

### Режим обработки и fake-ip

*   `enhanced-mode` — `fake-ip` или `redir-host` (по умолчанию `redir-host`): как DNS связан с маршрутизацией и подменой адресов.
*   `fake-ip-range` / `fake-ip-range6` — диапазоны IPv4/IPv6 для fake-ip (TUN часто завязан на тот же IPv4-диапазон).
*   `fake-ip-filter` — домены/маски/наборы, для которых **не** выдаётся fake-ip (см. [синтаксис масок](https://wiki.metacubex.one/en/handbook/syntax/#domain-wildcards)).
*   `fake-ip-filter-mode` — `blacklist` / `whitelist` / `rule`: в режиме **`rule`** список `fake-ip-filter` — это цепочка селекторов с политикой `fake-ip` / `real-ip` / `MATCH`, но ядро принимает **только доменные** типы правил (`DOMAIN*`, `GEOSITE`, `RULE-SET`, …). Строки вида **`IP-CIDR,...`** или **`GEOIP,...`** дают ошибку парсинга: *rule type 'IP-CIDR' not supported, only domain-based rules allowed*. Исключения по подсети IP для DNS так не задаются — используйте домены или переносите **`IP-CIDR`** в блок **`rules:`** (маршрутизация уже к известному IP после резолва).
*   `fake-ip-ttl` — TTL ответов fake-ip; менять только при необходимости.

### Цепочки серверов

*   `default-nameserver` — DNS **только для резолва имён самих DNS-серверов** из списков (должны быть достижимы без «курицы и яйца»).
*   `nameserver` — основные серверы для обычных запросов.
*   `fallback` — резервные (часто «зарубежные») серверы; при наличии `fallback` по умолчанию включается `fallback-filter` с `geoip-code: CN` и т.д.
*   `nameserver-policy` — политика «какой домен → каким сервером резолвить»; имеет приоритет над общими `nameserver`/`fallback`. Значения — строка или массив серверов; поддерживаются `rcode://…` (см. раздел [схем адресов](#sec-dns-schemes)).
*   `proxy-server-nameserver` / `proxy-server-nameserver-policy` — отдельные серверы **только для резолва доменов прокси-узлов** (имена в `proxies:`). Если пусто — используются обычные цепочки.
*   `direct-nameserver` / `direct-nameserver-follow-policy` — DNS для исходящего трафика в режиме DIRECT.

### Утечки и правила

*   `respect-rules: false` — при `true` запросы DNS к публичным адресам идут согласно [правилам `rules:`](https://wiki.metacubex.one/en/config/rules/); нужен осмысленный `proxy-server-nameserver`. С `prefer-h3` **совмещать не рекомендуется** (вики).
*   `use-hosts` / `use-system-hosts` — учитывать блок `dns.hosts` и системный hosts (см. раздел [DNS Hosts](#sec-dns-hosts)).

### `fallback-filter`

Условия «подмены» ответа основного `nameserver`, при которых берётся результат `fallback`: `geoip`, `geoip-code`, `geosite`, `ipcidr`, `domain`.

### Параметры в строке сервера (`#` и `&`)

Для публичных DNS в URI можно добавлять хвост вида `#proxy&h3=true&skip-cert-verify=false&ecs=…` — прокси/интерфейс для соединения, принудительный HTTP/3 для DoH, ECS, отключение IPv4/IPv6 и др. См. [Additional Parameters](https://wiki.metacubex.one/en/config/dns/#additional-parameters).

```yaml
dns:
  enable: true
  listen: 0.0.0.0:1053
  ipv6: false
  enhanced-mode: fake-ip
  use-hosts: true
  default-nameserver:
    - 223.5.5.5
  nameserver:
    - https://dns.alidns.com/dns-query
  fallback:
    - tls://8.8.4.4
  nameserver-policy:
    'geosite:cn':              # или ключи вида '+.example.com', см. вики
      - https://doh.pub/dns-query
```

## Поддерживаемые типы DNS в Mihomo (DNS Types)

В блоках `default-nameserver`, `nameserver` и `fallback` ядро Mihomo поддерживает работу со всеми существующими сетевыми протоколами. Тип протокола определяется префиксом (схемой) перед адресом сервера.

---

### 1. Базовые и шифрованные протоколы

*   `udp://` — классический незашифрованный DNS. Префикс `udp://` можно опускать (запись `8.8.8.8` по умолчанию считается UDP). Самый быстрый, но легко перехватывается и подменяется провайдером.
*   `tcp://` — DNS через TCP-соединение (например, `tcp://8.8.8.8`). Медленнее, чем UDP, но защищает от простейшего спуфинга (подмены пакетов "на лету").
*   `tls://` — DNS-over-TLS (DoT). Трафик полностью упаковывается в криптографический TLS-туннель (например, `tls://1.1.1.1`). Провайдер видит только факт подключения к серверу, но не знает, какие сайты вы запрашиваете.
*   `https://` — DNS-over-HTTPS (DoH). Самый популярный протокол (например, `https://dns.google`). Запросы маскируются под обычный HTTPS-трафик (веб-серфинг), из-за чего их практически невозможно заблокировать отдельно от остальных сайтов.

---

### 2. Продвинутые и специфические типы (Новая информация)

*   `quic://` — DNS-over-QUIC (DoQ). Современный шифрованный протокол (например, `quic://dns.adguard-dns.com` — уточняйте актуальный хост у провайдера DoQ). Сочетает свойства DoT/DoH со скоростью UDP.
*   `system://` — принудительное использование системных DNS-настроек самого роутера (запись `system://` или просто `system`). Позволяет Mihomo брать DNS напрямую из системного файла `/etc/resolv.conf` или операционной системы KeeneticOS.
*   `dhcp://` — динамический подхват DNS. Схема `dhcp://интерфейс` (например, `dhcp://en0`) заставляет Mihomo использовать DNS-сервера, которые роутер получил по DHCP от конкретного интернет-кабеля или провайдера. Запись `dhcp://system` дублирует системные DNS.

---

### 3. Виртуальный протокол ответов (rcode://)

Специальный тип правил `rcode://`, который используется внутри блока `nameserver-policy` для мгновенного возврата статических ответов клиенту БЕЗ отправки запроса в интернет:

*   `rcode://success` — вернуть статус "Успешно" (используется для заглушек или фильтрации).
*   `rcode://name_error` — вернуть статус "Домен не существует" (NXDOMAIN). Идеальный инструмент для глухой блокировки трекеров, рекламы и ботнетов прямо на уровне DNS-запроса (быстрее и легче, чем Null-Routing через `hosts:`).
*   `rcode://refused` — вернуть статус "Запрос отклонен" (Refused). Принудительный отказ в обслуживании для конкретных доменов.

## Доменное снайпирование (Domain Sniffing) в Mihomo

Блок **sniffer** отвечает за перехват трафика на сетевом уровне роутера. Он заглядывает внутрь шифрованных пакетов (без расшифровки самого контента), чтобы прочитать реальное имя домена (SNI/Host) и точно сопоставить его с правилами маршрутизации. 

Это критически важно при использовании режима `fake-ip` или когда приложения обращаются напрямую к серверам по IP-адресам без DNS-запроса.

---

### Глобальные параметры

*   `enable: true` — включение или выключение работы снайфера на роутере.
*   `force-dns-mapping: true` — принудительное сопоставление доменов для старого режима трафика `redir-host`.
*   `parse-pure-ip: true` — заставляет снайфер шпионить за абсолютно любым «чистым» IP-трафиком, у которого изначально не было обнаружено имя домена.
*   `override-destination: true` — перезаписывать ли оригинальный IP-адрес назначения тем реальным IP, который снайфер найдет после определения имени домена. По умолчанию рекомендуется `true`.

---

### Настройка протоколов (sniff)
*Ядро Mihomo поддерживает глубокий разбор только трех основных веб-протоколов:*

```yaml
sniffer:
  sniff:
    HTTP:
      ports: [80, 8080-8880]       # Порты для обычного незашифрованного веб-трафика
      override-destination: true   # Локальное переопределения правила для HTTP
    TLS:
      ports: [443, 8443]           # Порты для стандартного шифрованного HTTPS
    QUIC:
      ports: [443, 8443]           # Порты для современного трафика HTTP/3
```

---

### Тонкая фильтрация и исключения

*   `force-domain` — список доменов, которые снайфер обязан сканировать принудительно, даже если они нарушают другие правила. Поддерживает маски (`+.v2ex.com`).
*   `skip-domain` — список доменов, которые снайферу **запрещено** трогать. Сюда заносят сайты Рунета (`geosite:ru`), YouTube и Discord, чтобы снайфер не конфликтовал с сетевым обходом `nfqws2` (см. [раздел про nfqws2](#sec-nfqws)).
*   `skip-src-address` — белый список исходных IP-адресов внутри вашей домашней сети. Полезно, чтобы полностью запретить снайферу сканировать трафик конкретного умного телевизора, приставки или смартфона.
*   `skip-dst-address` — белый список целевых IP-адресов в интернете, трафик к которым снайфер должен игнорировать (например, локальные адреса провайдера или доверенные DNS-сервера).

---

### Пример идеальной конфигурации (без конфликтов с nfqws2)

Ниже — ориентир по `sniffer`; общая схема с **nfqws2** и порядком правил — в [отдельном разделе](#sec-nfqws).

```yaml
sniffer:
  enable: true
  force-dns-mapping: true
  parse-pure-ip: true
  override-destination: true
  sniff:
    TLS:
      ports: [443, 8443]
      skip-domain:
        - 'geosite:google'
        - 'geosite:discord'
        - 'geosite:youtube'
        - 'geosite:ru'
    HTTP:
      ports: [80, 8080-8880]
      override-destination: true
      skip-domain:
        - 'geosite:google'
        - 'geosite:discord'
        - 'geosite:ru'
    QUIC:
      ports: [443, 5055, 5056]
      skip-domain:
        - 'geosite:google'
        - 'geosite:discord'
        - 'geosite:ru'
```

## Входящие подключения (Inbounds / Listeners) в Mihomo

Этот блок настраивается внутри секции `listeners:` (или через старые глобальные параметры вроде `mixed-port`). Он определяет, какие порты и протоколы Mihomo открывает на роутере для приема трафика от ваших локальных устройств, а также позволяет превратить сам роутер в полноценный зашифрованный сервер.

Официальное описание **общих полей** каждой записи в списке: [LISTENERS (en)](https://wiki.metacubex.one/en/config/inbound/listeners/) · [LISTENERS (zh)](https://wiki.metacubex.one/config/inbound/listeners/). Полные примеры по типам: [Inbound configuration](https://wiki.metacubex.one/en/config/inbound/).

---

<a id="sec-listeners-fields"></a>

### 1. Общие поля записи в `listeners:`

Поля из [страницы LISTENERS](https://wiki.metacubex.one/en/config/inbound/listeners/) применимы ко **всем** типам inbound (если для конкретного `type` нет своих обязательных ключей вроде `password` у Shadowsocks-сервера).

*   **`name`** — уникальное имя слушателя. На него можно ссылаться в правилах [`IN-NAME`](https://wiki.metacubex.one/en/config/rules/#in-name).
*   **`type`** — вид inbound (`socks`, `http`, `mixed`, `redir`, `tproxy`, `tunnel`, `tun`, `shadowsocks`, `vmess`, `tuic` и др.); см. следующий подраздел.
*   **`listen`** — адрес привязки (например, `0.0.0.0` или `127.0.0.1`). В примерах вики по умолчанию подразумивается прослушивание на **`0.0.0.0`**, если `listen` не указан — уточняйте для своей версии ядра в логах/документации.
*   **`port`** — порт или **запись в формате `ports`** (несколько портов и диапазонов в одной строке). Синтаксис тот же, что для правил: диапазон через `-`, несколько групп через `/` или `,`. Пример из [вики — Port Ranges](https://wiki.metacubex.one/en/handbook/syntax/#port-ranges): `114-514/810-1919,65530` (порты 114–514, 810–1919 и 65530).
*   **`rule`** — имя набора из корневого блока `sub-rules:`. Если указанное подправило **не найдено**, для этого inbound используются обычные глобальные **`rules:`** ([пояснение в inbound](https://wiki.metacubex.one/en/config/inbound/)).
*   **`proxy`** — если не пусто, весь трафик с этого inbound сразу уходит на указанный **прокси или группу**; глобальные `rules:` для него не применяются. Имя **обязано** существовать в конфиге (`proxies` / `proxy-groups`), иначе ядро выдаст **ошибку** ([вики — proxy](https://wiki.metacubex.one/en/config/inbound/listeners/#proxy)).

*   **`udp`** — не на странице LISTENERS как отдельный заголовок, но в примерах [inbound](https://wiki.metacubex.one/en/config/inbound/): для многих типов можно задать `udp: false`; по умолчанию в примерах закомментировано **`udp: true`**.

---

### 2. Локальные inbounds (прозрачное проксирование и локальные порты)

Каждый слушатель в списке `listeners:` настраивается как отдельный элемент со своими уникальными свойствами:

*   `type: socks` — классический прокси-порт SOCKS5. Требует явной настройки в приложениях на ПК (например, в Telegram).
*   `type: http` — стандартный HTTP/HTTPS прокси-порт.
*   `type: mixed` — комбинированный порт (автоматически определяет HTTP или SOCKS5 запрос). Самый удобный вариант для ручной настройки программ.
*   `type: redir` — порт для прозрачного перехвата только TCP-трафика с помощью системного механизма `iptables REDIRECT`.
*   `type: tproxy` — порт для полноценного прозрачного проксирования `TPROXY` (обрабатывает и TCP, и тяжелый UDP трафик без потерь). Идеален для обхода блокировок в играх на консолях и ПК.
*   `type: tunnel` — порт-туннель: весь входящий трафик пересылается на фиксированный `target:` (см. также корневой блок [`tunnels:`](#sec-tunnels)).
*   `type: tun` — отдельный TUN-inbound в `listeners:` (отличается от корневого `tun:`): нужны `inet4-address` / `inet6-address`, для `stack` доступны варианты **`gvisor` и `lwip`** (см. [вики — inbound](https://wiki.metacubex.one/en/config/inbound/)).

---

### 3. Серверные inbounds (роутер как прокси-сервер)

Mihomo умеет работать «в обратную сторону». Вы можете открыть порты наружу, чтобы подключаться к своему домашнему роутеру (и использовать его интернет) из любой точки мира по зашифрованным протоколам.

*   `type: shadowsocks` — поднимает на роутере Shadowsocks-сервер. Требует настройки шифрования (`cipher: 2022-blake3-aes-256-gcm`) и пароля (`password`).
*   `type: vmess` — поднимает сервер VMess с авторизацией по списку UUID пользователей.
*   `type: tuic` — поднимает современный, сверхбыстрый сервер TUIC v4/v5 на базе HTTP/3 (QUIC). Идеально подходит для обхода жестких мобильных блокировок со смартфона через домашний роутер. Поддерживает алгоритм контроля перегрузок `congestion-controller: bbr`.

---

### 4. Комплексный пример конфигурации `listeners`

```yaml
listeners:
  # Наш основной смешанный порт для локальных программ
  - name: mixed-local
    type: mixed
    port: 10810
    listen: 0.0.0.0
    udp: true

  # Прозрачный TPROXY порт, куда XKeen заворачивает трафик сети
  - name: tproxy-routing
    type: tproxy
    port: 10812
    listen: 0.0.0.0
    udp: true

  # Принудительный туннель: всё, что попадет на порт 10816, улетит строго на VPS в Ригу
  - name: forced-vps-tunnel
    type: tunnel
    port: 10816
    network: [tcp, udp]
    target: google.com
    proxy: Riga-VLESS
```

## Общие параметры прокси-узлов (Proxies / Common Fields) в Mihomo

Этот блок описывается внутри массива `proxies:` в конфигурационном файле `config.yaml`. Здесь задаются параметры подключения к вашим удаленным серверам (VPS). Помимо уникальных настроек для каждого протокола (VLESS, Shadowsocks и др.), у любого прокси-узла есть набор общих фундаментальных параметров управления трафиком.

**Единый справочник по всем типам `proxies:` и полям (RU):** [**docs/ru/proxies.md**](docs/ru/proxies.md).

---

### 1. Базовые обязательные параметры (Core Fields)

*   `name: "Riga-VLESS"` — уникальное имя прокси-узла. Используется для отображения в веб-панелях управления и для привязки к группам маршрутизации.
*   `type: vless` — тип протокола шифрования (например: `ss`, `vmess`, `vless`, `trojan`, `hysteria2`, `tuic`).
*   `server: 94.158.219.192` — IP-адрес или доменное имя вашего удаленного VPS-сервера.
*   `port: 31237` — сетевой порт, на котором сервер принимает входящие подключения.

---

### 2. Управление сетевыми интерфейсами и протоколами

*   `ip-version: ipv4` — версия IP-протокола для исходящих соединений с сервером (если адрес сервера указан в виде домена). Доступные значения: `dual`, `ipv4`, `ipv6`, `ipv4-prefer`, `ipv6-prefer`. Для связки с `nfqws2` на роутерах рекомендуется жестко выставлять `ipv4` или `ipv4-prefer` ([почему](#sec-nfqws)).
*   `udp: true` — разрешение на пропуск UDP-трафика через данный прокси-узел. Для протоколов на базе QUIC (Hysteria2, TUIC), а также встроенных типов `direct` и `dns` этот параметр активен автоматически.
*   `interface-name: eth0` — принудительная привязка прокси-узла к конкретному физическому интерфейсу роутера (интернет-кабелю). Трафик к VPS пойдет строго через указанную сетевую карту.
*   `routing-mark: 1234` — накладывает цифровой маркер (fwmark) на исходящие пакеты к прокси-серверу на уровне ядра Linux.
*   `tfo: false` — включение механизма TCP Fast Open. Позволяет передавать полезные данные уже на этапе отправки первого пакета рукопожатия (SYN), значительно ускоряя установку соединения. Работает только для TCP.
*   `mptcp: false` — активация Multipath TCP. Позволяет одному прокси-соединению параллельно использовать несколько интернет-каналов (например, проводной интернет + 4G модем на роутере) для повышения скорости и отказоустойчивости.

---

### 3. Цепочки проксирования (dialer-proxy)

*   `dialer-proxy: имя_узла` — **«Матрешка из прокси»**. Позволяет установить соединение с текущим сервером не напрямую через вашего провайдера, а предварительно пропустив трафик через другой указанный прокси-узел или прокси-группу. Помогает строить сложные цепочки обхода («Провайдер -> Прокси 1 -> Прокси 2 -> Целевой сайт»).

---

### 4. Мультиплексирование соединений (smux / sing-mux)

Блок `smux:` активирует технологию слияния потоков. Вместо открытия сотен отдельных TCP-подключений для каждой картинки или скрипта на сайте, Mihomo упаковывает весь этот трафик внутрь **одного единственного** постоянного TCP-туннеля до вашего VPS. Это кардинально снижает нагрузку на слабый процессор домашнего роутера.

*   `enabled: true` — включение или выключение мультиплексирования для данного прокси.
*   `protocol: h2mux` — протокол слияния потоков. Доступные варианты: `smux`, `yamux`, `h2mux` (по умолчанию используется наиболее стабильный `h2mux`).
*   `max-connections: 4` — максимальное количество параллельных базовых TCP-туннелей, которые Mihomo может открыть до сервера при высокой нагрузке. Конфликтует с параметром `max-streams`.
*   `min-streams: 4` — минимальное количество мелких сетевых потоков внутри одного туннеля, после достижения которого ядро начнет открывать новое базовое TCP-соединение.
*   `max-streams: 0` — жесткий лимит на максимальное количество внутренних потоков в одном туннеле перед открытием нового (0 — без ограничений).
*   `statistic: false` — отображать ли базовые служебные туннели smux отдельно в веб-интерфейсе панели для возможности их ручного разрыва.
*   `only-tcp: false` — если выставлено `true`, настройки мультиплексирования не будут затрагивать UDP-пакеты (они пойдут напрямую по своим стандартным маршрутам).
*   `padding: true` — добавляет случайный мусорный размер (набивку пакетов) к служебным заголовкам мультиплексора, чтобы защитить структуру трафика от анализа системами DPI провайдеров.

---

### 5. Скоростной алгоритм TCP Brutal (brutal-opts)

Эксклюзивная подсистема для борьбы с жестким провайдерским зажиманием скорости и потерей пакетов на нестабильных магистральных каналах. Работает по принципу агрессивного заполнения полосы пропускания, заставляя провайдера отдавать вам приоритет по скорости.

*   `enabled: true` — активация алгоритма контроля перегрузок TCP Brutal для данного прокси-соединения.
*   `up: 50` — заявленная скорость отдачи (в Мегабитах в секунду, Mbps) вашего интернет-тарифа.
*   `down: 100` — заявленная скорость загрузки (в Мегабитах в секунду, Mbps) вашего интернет-тарифа. Ядро будет принудительно разгонять туннель до этих лимитов, игнорируя попытки DPI искусственно замедлить трафик.

## Группы прокси-серверов (Proxy Groups) в Mihomo

Этот блок настраивается внутри массива `proxy-groups:` в файле `config.yaml`. Группы объединяют ваши прокси-сервера (VPS) и встроенные политики (DIRECT, REJECT) в удобные логические узлы для автоматизации. Именно на имя конкретной группы ссылаются ваши правила в блоке `rules:`.

---

### 1. Фундаментальные параметры

*   `name: "Имя_Группы"` — уникальное имя группы. Если название содержит специальные символы, пробелы или кириллицу, его обязательно нужно оборачивать в одинарные кавычки (`'XKeen'` или `'Политика_Дискорда'`).
*   `type: select` — тип работы группы. Определяет, как Mihomo распределяет трафик между серверами внутри этой группы.
*   `proxies:` — список серверов и встроенных политик (`DIRECT`, `REJECT`), которые входят в эту группу.
*   `use:` — подключение внешних динамических списков прокси-серверов через модули `proxy-providers` (например, для автоматического скачивания пула серверов по подписке из внешнего файла).

---

### 2. Типы прокси-групп (Параметр type)

*   `select` — ручной выбор. Узел активирует строго тот прокси-сервер, на который вы лично кликнули мышкой в веб-интерфейсе панели управления XKeen-UI.
*   `url-test` — автоматический выбор по задержке: ядро периодически делает **HTTP(S)-запрос** к `url:` через каждый узел (это не ICMP-ping). Ответ **любого** успешного кода по умолчанию может считаться «живым»; точную проверку задают через `expected-status` (см. ниже).
*   `fallback` — резервное переключение (отказоустойчивость). Трафик всегда идет через самый первый сервер в списке. Если он внезапно падает, Mihomo мгновенно перекидывает поток на второй узел, затем на третий и так далее.
*   `load-balance` — балансировка нагрузки. Трафик распределяется между всеми серверами группы одновременно (например, по алгоритмам `round-robin` или `consistent-hash`) для достижения максимальной скорости.
*   `relay` — цепочка (ретрансляция). Трафик последовательно проходит через все указанные в списке узлы по очереди («Вход -> Сервер 1 -> Сервер 2 -> Выход»).

---

### 3. Автотесты и контроль доступности (Health Check)

*Эти настройки применяются для автоматических групп (`url-test`, `fallback`, `load-balance`):*

*   `url: 'http://www.gstatic.com/generate_204'` — URL для health check. Часто используют **`generate_204`**: сервер отвечает **`204 No Content`** маленьким телом — удобно для замера задержки. Если указать, например, только `https://www.gstatic.com` без пути, ответ может быть **200** с HTML; без настройки `expected-status` это всё равно может считаться успехом, но поведение и нагрузка отличаются от «лёгкого» 204 — при сомнениях задайте `expected-status` явно (см. следующий пункт).
*   `interval: 300` — интервал проведения автотестов в секундах. Значение `0` полностью отключает периодический пинг серверов.
*   `lazy: true` — **«Ленивый режим»**. Если узел или группа сейчас не используются текущими правилами маршрутизации, ядро полностью останавливает для них фоновые тесты скорости, разгружая процессор и экономя трафик. По умолчанию рекомендуется `true`.
*   `timeout: 5000` — максимальное время ожидания ответа от сервера при тесте доступности (в миллисекундах).
*   `max-failed-times: 5` — критическое число неудачных попыток опроса подряд, после которого Mihomo принудительно пометит сервер как «мертвый» и исключит его из маршрутизации.
*   `expected-status: 200/302/400-503` — проверка точного HTTP-кода ответа при тесте. Сервер будет считаться рабочим только в том случае, если код совпал с маской (поддерживает перечисление через косую черту и диапазоны через дефис).

---

### 4. Динамическая фильтрация и сортировка узлов

*Эти параметры используются для автоматического разбора тяжелых списков подписок (`proxy-providers`):*

*   `include-all: true` — автоматически добавляет в текущую группу вообще все прокси-сервера и сторонние провайдерские листы, отсортировав их по алфавиту.
*   `include-all-proxies: true` — автоматически добавляет в группу только жестко прописанные в конфиге одиночные прокси-узлы.
*   `include-all-providers: true` — автоматически добавляет в группу только прокси-узлы, полученные из внешних файлов подписок.
*   `filter: "(?i)riga|lv|latvia"` — одно регулярное выражение для отбора узлов по имени; несколько вариантов объединяйте внутри одного regex (`|` и т.д.).
*   `exclude-filter: "RU|Russia"` — регулярное выражение для исключения. Удаляет из группы сервера, содержащие в названии указанные ключевые слова.
*   `exclude-type: "Shadowsocks|Http"` — фильтрация по типу криптографии. Исключает из группы прокси-узлы, работающие на указанных протоколах.

---

### 5. Ограничения и кастомизация интерфейса API

*   `disable-udp: true` — принудительное отключение поддержки UDP на уровне всей прокси-группы (даже если у самих входящих в нее серверов UDP был разрешен).
*   `hidden: true` — скрывает отображение текущей прокси-группы в веб-панелях управления, разгружая интерфейс от служебных узлов (требует поддержки со стороны панели).
*   `icon: "https://example.com"` — кастомная иконка для красивого графического отображения прокси-группы внутри современных веб-интерфейсов управления (например, в панели Metacubexd).

---

### 6. Устаревшие параметры (Депрекация)
*Данные настройки признаны устаревшими и перенесены на уровень описания конкретных прокси-узлов. Использовать их внутри групп запрещено. Точная версия удаления — по [changelog](https://github.com/MetaCubeX/mihomo/releases) вашей ветки; при старте ядро может печатать предупреждение `deprecated` — ориентируйтесь на него и на свою версию `mihomo -v`.*
*   `interface-name` — принудительная привязка группы к физической сетевой карте.
*   `routing-mark` — накладывание цифрового маркера на исходящие сессии группы.

---

### Шаблон эталонной конфигурации блока `proxy-groups`

```yaml
proxy-groups:
  # 1. Основная группа ручного выбора
  - name: 'XKeen'
    type: select
    proxies:
      - 'Авто-Выбор-Скорости'        # Ссылка на соседнюю автоматическую группу
      - 'Riga-VLESS'
      - DIRECT

  # 2. Автоматическая группа: сама выбирает самый быстрый сервер каждые 5 минут
  - name: 'Авто-Выбор-Скорости'
    type: url-test
    proxies:
      - 'Riga-VLESS'
      - 'Backup-Shadowsocks'
    url: 'http://www.gstatic.com/generate_204'
    interval: 300
    lazy: true
    timeout: 5000
```

## Протокол VLESS в Mihomo (XTLS Vision & REALITY)

Протокол **VLESS** — это облегченный сетевой протокол без встроенного шифрования данных (поскольку само шифрование перекладывается на транспортный слой TLS). В ядре Mihomo он является основным инструментом для обхода жестких блокировок благодаря поддержке алгоритмов маскировки **XTLS Vision** и технологии **REALITY**.

---

### 1. Обязательные параметры (Core Fields)

*   `uuid` — уникальный идентификатор пользователя (User ID) в формате UUID. Служит ключом авторизации на вашем VPS-сервере.
*   `flow: xtls-rprx-vision` — подпротокол XTLS Vision (только когда сервер его поддерживает). Для обычного VLESS поверх TLS **без** Vision поле `flow` обычно **не указывают**.
*   `packet-encoding: xudp` — метод кодирования UDP-пакетов внутри VLESS-туннеля. Доступные значения: `packetaddr` (стандарт V2Ray 5+) или `xudp` (высокопроизводительный стандарт Xray). Для роутеров Keenetic **строго рекомендуется использовать `xudp`**, так как он критически важен для стабильной работы голосовых каналов Discord и онлайн-игр без потерь датаграмм.
*   `encryption: ""` — метод шифрования на стороне клиента. Для VLESS этот параметр **обязан оставаться пустым (`""`)**, так как защита данных обеспечивается внешним TLS-слоем.

---

### 2. Параметры маскировки TLS и REALITY

*   `tls: true` — глобальное включение шифрования TLS для туннеля. При использовании REALITY этот параметр также должен быть активен.
*   `servername: mirror.yandex.ru` — адрес SNI (Server Name Indication). Имя доверенного сайта-ширмы, под который ваш роутер будет маскировать подключение к заблокированному VPS для обмана систем DPI провайдера.
*   `client-fingerprint: chrome` — цифровой отпечаток браузера (TLS Fingerprint). Заставляет Mihomo имитировать структуру пакетов популярного браузера (доступны варианты: `chrome`, `firefox`, `safari`, `edge`). Это защищает соединение от активного зондирования цензорами.
*   `skip-cert-verify: false` — отключает проверку подлинности SSL-сертификата. При использовании REALITY и легальных серверов этот параметр должен быть `false`, чтобы не снижать безопасность.

---

### 3. Специфический блок REALITY (reality-opts)

Технология REALITY позволяет вашему VPS «украсть» TLS-сертификат у любого крупного разрешенного сайта (например, Яндекса, Apple или Microsoft). Провайдер видит, что вы просто открываете популярный легальный ресурс, хотя внутри этого соединения идет зашифрованный трафик прокси.

*   `public-key` — открытый криптографический ключ (X25519 Public Key), сгенерированный на вашем VPS-сервере. Необходим для первичного безопасного обмена ключами.
*   `short-id` — короткий шестнадцатеричный идентификатор сессии (Short ID), прописанный в настройках сервера (можно оставлять пустым `''`).

---

### 4. Продвинутая рандомизация длины пакетов (Продвинутый encryption)

В ядре Mihomo параметр `encryption` получил расширенную логику для борьбы с ИИ-анализаторами трафика, которые вычисляют прокси по фиксированной длине пакетов при рукопожатии (handshake length characteristics).

Синтаксис выглядит как сложная строка параметров:
`encryption: "mlkem768x25519plus.native/xorpub/random.1rtt..."`

Она включает алгоритмы **Padding (Добавочный мусор)**:
*   После отправки клиентского запроса, ядро со 100% вероятностью дописывает к пакету случайный мусорный хвост размером от 111 до 1111 байт.
*   С вероятностью 75% выдерживается случайная пауза от 0 до 111 миллисекунд перед следующим пакетом.
*   Затем с вероятностью 50% отправляется вторая порция случайного шума до 3333 байт. 
*   *Это полностью ломает любые статистические модели анализа трафика на ТСПУ провайдеров.*

---

### 5. Транспортный уровень (network)

*   `network: tcp` — тип транспортного протокола для передачи данных. Помимо стандартного `tcp`, VLESS поддерживает упаковку трафика в альтернативные транспорты: `ws` (WebSockets), `http`, `h2` (HTTP/2), `grpc` или `xhttp`.

---

### Важное примечание разработчиков ядра (UDP block)
Реализация контроля потока `xtls-*` в Mihomo эквивалентна режиму `xtls-*-udp443` в Xray-core. Если вам необходимо полностью заблокировать прохождение UDP-пакетов по порту 443 (чтобы принудительно отключить протокол QUIC в браузерах), используйте логическое правило в блоке правил маршрутизации:
`- AND,((NETWORK,UDP),(DST-PORT,443)),REJECT`

---

### Эталонный шаблон конфигурации узла VLESS REALITY

```yaml
proxies:
  - name: 'Riga-VLESS'
    type: vless
    server: 94.158.219.192
    port: 31237
```

<a id="sec-rules"></a>

## Правила маршрутизации (Routing Rules) в Mihomo

Этот блок настраивается внутри массива `rules:` в конфигурационном файле `config.yaml`. Правила обрабатываются ядром строго сверху вниз. Первое совпавшее правило мгновенно отправляет трафик на указанную политику (DIRECT, REJECT, название прокси-группы), а все строки ниже полностью игнорируются.

---

### 1. Правила фильтрации доменов (Domain Matching)

*   `DOMAIN,ad.com,REJECT` — точное совпадение доменного имени. Сработает только на указанный адрес.
*   `DOMAIN-SUFFIX,google.com,XKeen` — совпадение по суффиксу. Покрывает `google.com`, `www.google.com`, `mail.google.com` и т.д., но **не** `content-google.com` (см. [вики](https://wiki.metacubex.one/en/config/rules/#domain-suffix)).
*   `DOMAIN-KEYWORD,google,XKeen` — поиск по ключевому слову. Сработает, если слово "google" содержится в любой части доменного имени.
*   `DOMAIN-WILDCARD,*.google.com,XKeen` — маска подстановочных знаков. Поддерживает символы `*` (ноль или более символов) и `?` (строго один символ). Отличается от классического формата Clash.
*   `DOMAIN-REGEX,^abc.*com,XKeen` — фильтрация доменов с использованием полноценных регулярных выражений (RegEx).
*   `GEOSITE,youtube,DIRECT` — сопоставление со встроенной базой категорий доменов `geosite.dat`. Самый экономичный способ группировки тысяч доменов одного сервиса.

---

### 2. Правила фильтрации по IP-адресам назначения (Destination IP)

*   `IP-CIDR,127.0.0.0/8,DIRECT` — проверка попадания IP-адреса назначения в указанную подсеть IPv4.
*   `IP-CIDR6,2620:0:2d0:200::7/32,XKeen` — проверка попадания IP-адреса назначения в указанную подсеть IPv6 (является полным аналогом IP-CIDR).
*   `IP-SUFFIX,8.8.8.8/24,XKeen` — совпадение по суффиксу (окончанию) IP-адреса.
*   `IP-ASN,13335,DIRECT` — фильтрация по номеру автономной системы (ASN) целевого сервера. Позволяет за секунду завернуть или заблокировать вообще все IP-адреса, принадлежащие конкретному провайдеру или корпорации (например, Cloudflare, Google, Ростелеком).
*   `GEOIP,CN,DIRECT` — проверка целевого IP по встроенной базе данных стран `geoip.dat`. Если IP принадлежит указанной стране (двухбуквенный код), правило срабатывает.

---

### 3. Правила фильтрации по параметрам источника (Source Filters)

*   `SRC-GEOIP,cn,DIRECT` — определяет страну происхождения источника (клиентского устройства) по базе GeoIP.
*   `SRC-IP-ASN,9808,DIRECT` — определяет номер автономной системы (ASN) устройства-источника трафика.
*   `SRC-IP-CIDR,192.168.1.201/32,DIRECT` — жесткая привязка правила к конкретному локальному IP-адресу вашего устройства (например, пустить трафик только вашего ТВ мимо прокси).
*   `SRC-IP-SUFFIX,192.168.1.201/8,DIRECT` — совпадение по суффиксу IP-адреса источника.

---

### 4. Сетевые порты и протоколы (Ports & Networks)

*   `DST-PORT,80,DIRECT` — фильтрация по целевому порту назначения (порту сайта/сервера). Поддерживает диапазоны (`50000-65535`).
*   `SRC-PORT,7777,DIRECT` — фильтрация по исходящему порту, который открыло приложение на вашем локальном компьютере.
*   `NETWORK,udp,DIRECT` — фильтрация по типу транспортного протокола (доступны значения `tcp` или `udp`).
*   `DSCP,4,DIRECT` — совпадение по полю DSCP в IP-заголовке; актуально в первую очередь для **входящего UDP через tproxy**. См. [вики — DSCP](https://wiki.metacubex.one/en/config/rules/#dscp).

---

### 5. Фильтрация входящих шлюзов Mihomo (Inbound Metadata)

*   `IN-PORT,7890,XKeen` — срабатывает, если трафик пришел на конкретный локальный порт Mihomo (например, на порт SOCKS прокси). Поддерживает диапазоны портов.
*   `IN-TYPE,SOCKS/HTTP,XKeen` — фильтрация по типу входящего протокола.
*   `IN-USER,mihomo,XKeen` — фильтрация по имени пользователя (актуально, если на портах HTTP/SOCKS настроена многопользовательская авторизация с паролями). Несколько имен разделяются косой чертой (`/`).
*   `IN-NAME,ss,XKeen` — фильтрация по уникальному имени входящего слушателя (параметр `name:` из блока `listeners:`).

---

### 6. Фильтрация процессов и ОС (Process & User ID)

*Внимание: Эти параметры работают только на тех устройствах, где Mihomo запущен локально (Windows, Android, Linux-сервер). На домашних роутерах Keenetic их использовать бесполезно (`find-process-mode: off`), так как роутер не видит имена процессов удаленных ПК.*

*   `PROCESS-PATH,/usr/bin/wget,XKeen` — проверка по полному системному пути к исполняемому файлу программы (работает на Windows и Linux).
*   `PROCESS-PATH-WILDCARD,/usr/*/wget,XKeen` — wildcard-маска для путей файлов (поддерживает `*` и `?`).
*   `PROCESS-PATH-REGEX,(?i).*Application\\chrome.*,XKeen` — регулярное выражение для путей процессов.
*   `PROCESS-NAME,chrome.exe,XKeen` — фильтрация по имени процесса. На платформе Android этот параметр автоматически сопоставляется с именем пакета приложения (`com.termux`).
*   `PROCESS-NAME-WILDCARD,*telegram*,XKeen` — wildcard-маска для имен процессов и пакетов Android.
*   `PROCESS-NAME-REGEX,(?i)Telegram,XKeen` — регулярное выражение для имен процессов.
*   `UID,1001,DIRECT` — фильтрация по цифровому идентификатору пользователя (User ID) в операционных системах Linux.

---

### 7. Логические операторы (AND, OR, NOT)

Позволяют создавать сложнейшие комбинированные условия. При их написании необходимо **строго следить за парными круглыми скобками**.

*   `AND,((DOMAIN,baidu.com),(NETWORK,UDP)),DIRECT` — оператор «И». Правило сработает только тогда, когда выполняются **оба** условия одновременно (домен baidu.com И протокол UDP).
*   `OR,((NETWORK,UDP),(DOMAIN,baidu.com)),REJECT` — оператор «ИЛИ». Правило сработает, если выполняется **хотя бы одно** из указанных условий.
*   `NOT,((DOMAIN,baidu.com)),XKeen` — оператор «НЕ». Инвертирует правило. Сработает для абсолютно любого трафика, кроме домена baidu.com.

---

### 8. Служебные селекторы (Rule-Sets, Sub-Rules, Match)

*   `RULE-SET,antifilter,XKeen` — подключение внешних тяжелых списков правил (Rule-Providers). Позволяет не забивать основной конфиг миллионами строк, а читать их из внешнего файла.
*   `SUB-RULE,(NETWORK,tcp),sub-rule-name` — перенаправление трафика во вложенный изолированный под-список правил (`sub-rule`). Требует осторожного использования круглых скобок.
*   `MATCH,DIRECT` — финальное безусловное правило. Ловит абсолютно весь трафик, который не подошел ни под одну строчку выше. Обязано находиться в самом низу блока `rules:`.

---

### 9. Дополнительные параметры правил (Additional Parameters)

Дописываются в самом конце строки правила через запятую:

*   `no-resolve` — предотвращает запуск DNS-резолва. Применяется для правил, проверяющих целевые IP (`IP-CIDR`, `GEOIP`, `IP-ASN`). По умолчанию, когда домен натыкается на IP-правило, Mihomo принудительно запрашивает DNS, чтобы узнать IP сайта и сравнить его с правилом. Флаг `no-resolve` заставляет ядро пропустить этот шаг (если домен не был зарезолвлен ранее, правило просто проигнорируется). Помогает избегать утечек DNS и ускоряет работу.
*   `src` — принудительная инверсия. Переводит сопоставление целевого IP-адреса назначения (Target IP) на сопоставление с IP-адресом источника (Source IP). Работает только для IP-правил.

В [официальной вики](https://wiki.metacubex.one/en/config/rules/#additional-parameters) для стандартного Mihomo перечислены именно эти два хвоста. Расширения вроде **`no-log`** встречаются в **сторонних сборках** (например, Mihomo+ / обвязка роутера): на «ванильном» ядре лишний токен может дать **ошибку парсинга** или быть проигнорирован — смотрите лог при старте.

#### Типичные ошибки в `rules:` и в `fake-ip-filter` (режим `rule`)

*   **`GEOIP`** — сопоставляет только **код страны** из GeoIP (двухбуквенный ISO, например `RU`, `CN`, `US`). Строки вида `GEOIP,discord`, `GEOIP,chatgpt`, `GEOIP,cisco`, `GEOIP,antifilter` **не соответствуют** документации: такие правила не делают то, что ожидается по имени. Для сервисов используйте **`GEOSITE`**, **`DOMAIN`/`DOMAIN-SUFFIX`**, **`RULE-SET`** или **`IP-CIDR`/`IP-ASN`**.
*   **`DOMAIN`** — ожидается **имя хоста** (FQDN), а не IPv4/IPv6 и не CIDR. Для адресов и подсетей в **`rules:`** — **`IP-CIDR`** / **`IP-CIDR6`** (и при необходимости **`no-resolve`**).
*   В **`fake-ip-filter`** при **`fake-ip-filter-mode: rule`** допустимы только **доменные** селекторы с политикой `fake-ip` / `real-ip` / **`MATCH`**. **`IP-CIDR` / `GEOIP` / `IP-ASN`** сюда подставлять нельзя — конфиг не поднимется (см. сообщение ядра про *only domain-based rules allowed*).

<a id="rule-provider-formats"></a>

### 10. Форматы содержимого файлов Rule-Providers

Внешние файлы, которые скачивает или читает секция **`rule-providers:`** в `config.yaml` (см. [Rule Providers](https://wiki.metacubex.one/en/config/rule-providers/)), при использовании типов `classical` / `domain` / `ipcidr` содержат внутри массив `payload:`. Синтаксис строк внутри `payload` должен соответствовать выбранному типу провайдера.

---

### А. Тип Classical (Классический многоцелевой формат)

Формат `classical` является самым универсальным. Он поддерживает абсолютно любые стандартные селекторы правил маршрутизации (кроме рекурсивного вызова других `rule-set` и `sub-rule`). Идеально подходит для комплексных кастомных списков обхода.

Пример структуры файла:
```yaml
payload:
  - DOMAIN-SUFFIX,google.com
  - DOMAIN-KEYWORD,google
  - DOMAIN,ad.com
  - SRC-IP-CIDR,192.168.1.201/32
  - IP-CIDR,127.0.0.0/8
  - GEOIP,CN
  - DST-PORT,80
  - SRC-PORT,7777
```

---

### Б. Тип Domain (Высокоскоростной текстовый формат доменов)

Формат `domain` предназначен исключительно для хранения списков доменных имен. Его главное преимущество — ядро Mihomo компилирует этот список в оперативной памяти в виде высокоскоростного дерева префиксов (Trie), что позволяет мгновенно проверять миллионы доменов без нагрузки на процессор.

*Важно:* Синтаксис масок внутри этого файла подчиняется классическим правилам wildcards формата оригинального Clash, а не Mihomo.

Пример структуры файла:
```yaml
payload:
  - '.blogger.com'              # Покрывает blogger.com и все его поддомены
  - '*.*.microsoft.com'          # Точечная wildcard-маска для поддоменов третьего уровня
  - 'apple.com'                    # Точное совпадение домена (корень)
```

---

### В. Тип Ipcidr (Формат списков сетевых диапазонов)

Формат `ipcidr` используется для создания списков IP-адресов и целых подсетей (как IPv4, так и IPv6). Mihomo оптимизирует чтение этого файла с помощью алгоритма Radix Tree, что критически важно для работы с тяжелыми списками (например, диапазонами заблокированных IP-адресов от Антифильтра).

*Важно:* Внутри этого файла пишутся исключительно IP-адреса с масками подсетей, никакие текстовые команды или названия доменов здесь недопустимы.

Пример структуры файла:
```yaml
payload:
  - '192.168.1.0/24'             # Диапазон локальной или целевой подсети
  - '10.0.0.1/32'                # Точечный одиночный IP-адрес конкретного сервера
  - '2001:db8::/32'              # Подсеть протокола IPv6
```

## Провайдеры прокси (Proxy Providers) в Mihomo

Этот блок настраивается внутри секции `proxy-providers:` в файле `config.yaml`. Он предназначен для динамического скачивания, автоматического обновления и гибкой фильтрации пулов прокси-серверов из внешних источников (например, готовых подписок на сторонние VLESS/Shadowsocks сервера).

---

### 1. Базовые параметры управления (Core Fields)

*   `type` — тип источника провайдера (обязательный параметр). Доступные варианты: `http` (скачивание по ссылке), `file` (чтение локального файла) или `inline` (описание серверов прямо внутри этого блока).
*   `url` — прямая ссылка на подписку. Требуется настроить, только если параметр `type` выставлен в значение `http`.
*   `path: ./proxy_providers/file.yaml` — уникальный путь для сохранения файла подписки. По умолчанию Mihomo сохраняет файлы внутри своего каталога (`HomeDir`), используя хэш MD5 от ссылки в качестве имени. Использование внешних путей за пределами домашней папки заблокировано из соображений безопасности.
*   `interval: 3600` — интервал автоматического перескачивания и обновления файла подписки в секундах (3600 = 1 час).
*   `proxy: DIRECT` — имя прокси-узла или группы, через которую ядро должно скачивать данный файл подписки. Если провайдер заблокирован, можно указать ваш рабочий VPS (например, `proxy: Riga-VLESS`).
*   `size-limit: 0` — ограничение на максимальный размер скачиваемого файла подписки в байтах. Значение `0` полностью отключает лимиты на размер.
*   `header` — кастомные HTTP-заголовки запроса при скачивании (например, можно вручную передать User-Agent или токен авторизации).

---

### 2. Встроенный контроль доступности (Health Check)

Позволяет ядру автоматически пинговать и отсеивать нерабочие прокси-сервера, полученные из подписки, еще до того, как они попадут в основные группы маршрутизации.

*   `health-check.enable: true` — включение автоматического теста задержки для серверов текущего провайдера.
*   `health-check.url: https://gstatic.com` — адрес тестовой страницы. Рекомендуется использовать стабильные endpoints от Google или Cloudflare.
*   `health-check.interval: 300` — интервал проверки серверов в секундах.
*   `health-check.timeout: 5000` — таймаут ожидания ответа при тесте доступности в миллисекундах.
*   `health-check.lazy: true` — «ленивый» режим. Тесты полностью останавливаются, если ни один сервер из этой подписки сейчас не используется правилами роутера.
*   `health-check.expected-status: 204` — ожидаемый код HTTP-ответа, подтверждающий успешную работу узла.

---

### 3. Переопределение параметров узлов (Override)

Позволяет на лету модифицировать, принудительно заменять или добавлять параметры к серверам, полученным из подписки, без необходимости править сам скачанный файл.

*   `override.additional-prefix: "Текст | "` — автоматически добавляет указанный фиксированный текст в самое начало имени каждого сервера подписки.
*   `override.additional-suffix: " | Текст"` — добавляет фиксированный текст в самый конец имени серверов.
*   `override.proxy-name` — динамическая замена частей имени серверов с помощью регулярных выражений (RegEx). Параметр `pattern` ищет совпадение, а `target` заменяет его на нужный текст.
*   `override.udp: true` — принудительно включает поддержку UDP-трафика для всех скачанных серверов подписки.
*   `override.tfo: false` / `override.mptcp: false` — управление сетевыми механизмами TCP Fast Open и Multipath TCP.
*   `override.skip-cert-verify: true` — принудительное отключение проверки SSL-сертификатов для серверов.
*   `override.ip-version: ipv4-prefer` — задает приоритет версии протокола IP при резолве адресов серверов подписки.

---

### 4. Фильтрация и сортировка серверов подписки

*   `filter: "(?i)riga|lv|latvia"` — одно регулярное выражение для отбора по имени узла; несколько условий — внутри одного шаблона (`|`).
*   `exclude-filter: "DE|Germany"` — регулярное выражение для исключения серверов по названию из текущего пула провайдера.
*   `exclude-type: "ss|http"` — исключение по типу протокола шифрования. Позволяет полностью выкинуть из работы устаревшие или небезопасные типы прокси-серверов подписки. Регулярные выражения здесь не поддерживаются, типы перечисляются строго через вертикальную черту `|`.

---

### 5. Резервный режим (payload)

*   `payload` — массив со списком жестко прописанных вручную серверов внутри конфига. Работает в двух случаях: когда параметр `type` изначально выставлен в `inline`, либо как автоматический аварийный бэкап, если ядро Mihomo не смогло скачать файл по HTTP-ссылке или прочитать его с локального диска.

---

### Комплексный пример конфигурации `proxy-providers`

```yaml
proxy-providers:
  premium-subscription:
    type: http
    url: "https://example-provider.com"
    path: ./proxy_providers/premium_list.yaml
    interval: 3600
    proxy: DIRECT
    header:
      User-Agent:
        - "mihomo/1.18.3"
    health-check:
      enable: true
      url: https://gstatic.com
      interval: 300
      timeout: 5000
      lazy: true
    override:
      udp: true
      skip-cert-verify: true
      additional-prefix: "VPS | "
    filter: "(?i)riga|lv|latvia"        # Берем только Латвию
    exclude-type: "ss|http"             # Отсекаем Shadowsocks и HTTP прокси
```


<a id="sec-sub-rules"></a>

## Вложенные под-правила (Sub-Rules) в Mihomo

Блок **sub-rules** — это подсистема создания изолированных, независимых списков правил маршрутизации. Они работают как «функции» или «подпрограммы» внутри основного сетевого стека Mihomo. Вместо того чтобы перегружать глобальный блок `rules:`, вы можете вынести специфическую логику во вложенные списки.

---

### 1. Как устроен синтаксис

Под-правила описываются в корневом блоке `sub-rules:` в виде именованных списков. Внутри каждого списка можно использовать абсолютно любые стандартные селекторы правил (`DOMAIN`, `IP-CIDR`, `GEOIP` и т.д.), включая финальное правило `MATCH`.

Пример описания в конфиге:
```yaml
sub-rules:
  # Первое под-правило (например, для фильтрации китайских сайтов)
  rule1:
    - DOMAIN-SUFFIX,baidu.com,DIRECT
    - MATCH,PROXY

  # Второе под-правило (например, жесткая фильтрация DNS или админки)
  sub-rule2:
    - IP-CIDR,1.1.1.1/32,REJECT
    - IP-CIDR,8.8.8.8/32,ss1
    - DOMAIN,alidns.com,REJECT
```

---

### 2. Сценарии применения (Actionable Use Cases)

Вы можете вызывать созданные под-правила в двух ключевых местах конфигурации:

#### Сценарий А: Вызов из глобального блока `rules:` (Логические цепочки)
Позволяет отправить трафик на проверку во вложенный список только в том случае, если выполнено базовое условие. Это аналог конструкции `if/then` в программировании.
```yaml
rules:
  # Если тип трафика — TCP, отправить его на детальную проверку внутрь sub-rule2
  - SUB-RULE,(NETWORK,tcp),sub-rule2
  
  # Обычные глобальные правила
  - GEOSITE,google,XKeen
  - MATCH,DIRECT
```

#### Сценарий Б: Привязка к конкретному порту в блоке `listeners:`
Вы можете заставить конкретный входящий порт роутера обрабатывать трафик по совершенно уникальным правилам, полностью игнорируя глобальный блок `rules:`.
```yaml
listeners:
  - name: guest-socks
    type: socks
    port: 10890
    # Весь трафик, пришедший на этот порт, будет фильтроваться строго по логике rule1
    rule: rule1 
```
*Это идеально подходит для создания изолированных «гостевых» портов или портов для детского интернета с жестким черным списком сайтов.*

---

### 3. Главное преимущество под-правил
Использование `sub-rules` кардинально снижает нагрузку на процессор роутера. Вместо того чтобы гонять *каждый* пакет по огромному общему списку из тысяч правил, Mihomo мгновенно отсекает ненужные типы трафика (например, по протоколу или порту) и отправляет пакет на обработку в короткий, специализированный список.



<a id="sec-tunnels"></a>

## Прямые туннели пересылки (Tunnels) в Mihomo

Блок **tunnels** предназначен для создания портов сквозного форвардинга (пересылки) трафика. Он открывает локальный порт на роутере и принудительно перенаправляет абсолютно все прилетающие на него TCP и UDP пакеты на один жестко заданный целевой сервер в интернете. 

Этот процесс происходит в обход глобального блока правил `rules:`, а сам трафик при этом может дополнительно упаковываться внутрь выбранного прокси-туннеля.

---

### 1. Синтаксис и форматы записи

Ядро Mihomo поддерживает два равнозначных типа описания туннелей в конфигурационном файле: **однострочный** (для быстрой и компактной записи) и **многострочный** (для наглядной конфигурации сложных параметров).

#### А. Однострочный формат (Single Line)
Параметры записываются через запятую в строгом порядке: `протокол,локальный_адрес:порт,целевой_адрес:порт,имя_прокси`

```yaml
tunnels:
  - tcp/udp,127.0.0.1:6553,8.8.8.8:53,Riga-VLESS
```

#### Б. Многострочный формат (Multiple Lines)
Развернутая структура с явным объявлением каждого ключа:

```yaml
tunnels:
  - network: [tcp, udp]          # Протоколы, которые слушает туннель
    address: 127.0.0.1:6553      # Локальный порт прослушивания на роутере
    target: 8.8.8.8:53           # Финальная точка назначения в интернете
    proxy: Riga-VLESS            # Прокси-узел, через который пойдет поток
```

---

### 2. Подробный разбор параметров (Fields)

*   `network` — тип отслеживаемой сети и протоколов. Допускается указывать одиночный протокол `tcp` или `udp`, либо слушать их одновременно в формате массива `[tcp, udp]` или строки `tcp/udp`.
*   `address` — локальный сетевой интерфейс и порт на роутере, который займет Mihomo под этот туннель. Если указать `127.0.0.1:порт`, туннель будет доступен только внутри самого роутера. Если указать `0.0.0.0:порт`, к туннелю сможет подключиться любое устройство из вашей домашней локальной сети.
*   `target` — финальный адрес назначения в интернете, куда нужно доставить пакеты. Поддерживает запись в виде IP-адреса (`8.8.8.8:53`) или в виде текстового доменного имени (`target.com:443`).
*   `proxy` — необязательный (опциональный) параметр. Имя конкретного прокси-узла или прокси-группы, через которую пойдет пересылка. Если удалить этот параметр, трафик туннеля пойдет в интернет напрямую через вашего провайдера (**DIRECT**).

---

### 3. Практические сценарии использования (Use Cases)

*   **Защита и ускорение DNS (DNS Tunneling):** 
    Вы можете открыть на роутере локальный порт `127.0.0.1:6553`, нацелить его на сервер Google DNS `8.8.8.8:53` и пустить через стабильный `Riga-VLESS`. Теперь, если перенаправить системный DNS роутера на этот порт, все ваши DNS-запросы пойдут в зашифрованном виде внутри прокси-туннеля, полностью защищенные от перехвата (DNS-Hijacking) вашим провайдером.
*   **Проброс игровых серверов или баз данных:** 
    Быстрое создание защищенного сквозного моста до конкретного удаленного игрового сервера или рабочей базы данных через стабильный прокси с низким пингом, минуя общие системные правила маршрутизации.



<a id="sec-nfqws"></a>

## Совместная работа Mihomo и nfqws2 (Keenetic / Linux)

**nfqws2** (ветка **zapret**) работает на уровне **очереди netfilter (NFQUEUE)**: подмешивает/искажает пакеты так, чтобы обойти DPI у провайдера. **Mihomo** решает уже прикладную задачу — **куда отправить поток** (DIRECT, прокси, REJECT), часто после того как трафик попал в **tproxy/redir** или в **TUN**. Обе системы могут работать параллельно, если **не заставлять их бороться за одно и то же действие** (двойная подмена адреса/домена, двойной захват маршрута по умолчанию, лишний QUIC-туннель там, где его «ломает» только nfqws).

### 1. Разделение ролей

| Слой | nfqws2 | Mihomo |
|------|--------|--------|
| Обход ТСПУ на провайдере | да (десинхронизация, фрагментация и т.п.) | нет |
| Выбор исходящей политики (прокси / прямой выход) | нет | да (`rules:`, `proxy-groups`) |
| Подмена SNI/Host «для правил» | нет | да (`sniffer`, `fake-ip`) |

Имеет смысл явно решить: **какие протоколы и сервисы** вы доверяете **только nfqws** (обычно отдельные UDP/TCP потоки к известным CDN), а какие идут **через Mihomo** без лишнего вмешательства sniffer/fake-ip.

### 2. Рекомендуемые настройки ядра (`general`)

*   **`ipv6: false`** — типичная рекомендация для связки с zapret/nfqws2: стек IPv6 на роутере часто не участвует в той же цепочке обхода, а лишние AAAA-запросы усложняют картину.
*   **`find-process-mode: off`** — на роутере процессы клиентских ПК не видны; снижает нагрузку на CPU.

### 3. DNS и fake-ip

*   Для доменов, которые **nfqws целит по имени/IP из реального резолва**, в Mihomo обычно задают **`real-ip`** в `fake-ip-filter` (режим `rule` / `blacklist`) или **DIRECT** в `rules:` раньше общих правил на прокси — чтобы Mihomo **не подменял** ответ DNS искусственным `198.18.x.x`, пока nfqws строит сессию к настоящему узлу.
*   **`nameserver-policy`** для зон Рунета / игр / мессенджеров на «честный» DNS (как у вас для Yandex) хорошо сочетается с nfqws: меньше сюрпризов при первом коннекте.
*   Избегайте в `fake-ip-filter` конструкций вида **`DOMAIN,<IPv4>`** или **`DOMAIN,<CIDR>`** — это неверный тип для `DOMAIN`. В режиме **`rule`** туда же **нельзя** писать **`IP-CIDR,...`** (ошибка парсинга): фильтр fake-ip работает по **имени из DNS-запроса**, а не по списку целевых подсетей. Нужные подсети — в **`rules:`** (`IP-CIDR,... DIRECT` и т.д.).

### 4. Sniffer

*   В **`skip-domain`** стоит заносить зоны, где **не нужна** подмена назначения после анализа SNI/QUIC: **`geosite:ru`**, часто **`geosite:google`**, **`geosite:youtube`**, **`geosite:discord`**, хосты игровых сетей (**Photon**, **Steam voice** и т.д.) — иначе sniffer может **расходиться с тем, что уже «подправил» nfqws**.
*   При необходимости используйте **`skip-src-address`** для конкретных клиентов в LAN, если только им нужен «чистый» обход без sniffer.

### 5. Правила `rules:` и порядок строк

*   Потоки, которые **должны уйти в nfqws до прокси**, обычно описывают **выше** общих `GEOSITE`/`GEOIP` на прокси: типичный приём — **`DIRECT`** для нужных доменов/UDP и узкие **`AND`** (например, Discord UDP в DIRECT), затем уже остальной трафик на **`XKeen`**.
*   Если вы **режете QUIC в Mihomo** (`REJECT` UDP к `GEOSITE,google` и т.п.), убедитесь, что это **не ломает** сценарий, где nfqws обрабатывает **тот же** QUIC; иногда достаточно одного механизма.
*   Для исходящих узлов с доменом в **`server:`** задайте **`ip-version: ipv4`** (или **`ipv4-prefer`**) — меньше обходов по IPv6, которые nfqws мог не учитывать.

### 6. TUN Mihomo и маршрутизация

*   Корневой **`tun:`** с **`auto-route: true`** и **`strict-route: true`** перехватывает **маршрут по умолчанию** на самом роутере. Цепочка **nftables → nfqws → tproxy-порт Mihomo** на Keenetic часто строится **без** полного TUN-захвата; включение TUN поверх того же трафика легко даёт **петли, утечки или обход nfqws**. Включайте TUN только если понимаете **полную схему** маршрутов и mark/`routing-mark`.
*   Классическая схема на роутере: **прозрачный порт Mihomo** (`tproxy-port` / `listeners` type `tproxy`) + правила XKeen/nftables, **без** конфликтующего второго «глобального захвата».

### 7. Отладка

*   При сбоях смотрите по очереди: лог **nfqws**, счётчики **nftables**, затем лог Mihomo (**`log-level: debug`** кратковременно). Симптом «работает по IP, не работает по домену» чаще связан с **DNS/fake-ip/sniffer**, а не с nfqws.

---

<a id="sec-ntp"></a>

## Синхронизация времени (NTP) в Mihomo

Блок **ntp** отвечает за запуск встроенной службы синхронизации точного времени (Network Time Protocol) внутри самого ядра Mihomo. Точное время жизненно необходимо для проверки валидности криптографических TLS-сертификатов, генерации временных токенов авторизации (протоколы VMess/VLESS) и успешного прохождения рукопожатий (Handshake) с вашим удаленным VPS-сервером.

---

### 1. Подробный разбор параметров (Fields)

*   `enable: true` — глобальное включение или выключение встроенной службы NTP внутри ядра Mihomo.
*   `write-to-system: true` — принудительная синхронизация полученного точного времени с системными часами самого устройства. 
    *   *Важно:* Чтобы этот параметр успешно работал на роутере Keenetic или ПК, ядро Mihomo должно быть запущено с правами суперпользователя (`root` в Linux/Entware или Режим Администратора в Windows).
*   `server: time.apple.com` — хост или IP NTP-сервера (часто используют `pool.ntp.org`, `time.google.com` и т.п.). Указывайте реальный NTP-хост, а не URL со схемой `http://`.
*   `port: 123` — сетевой порт для отправки NTP-запросов. Стандартным общемировым портом для синхронизации времени является `123` (протокол UDP). Изменять его без специфической необходимости на стороне сервера запрещено.
*   `interval: 30` — временной интервал между автоматическими сессиями сверки времени. Указывается строго в **минутах**. По умолчанию ядро обновляет время каждые 30 минут.

---

### 2. Практический пример конфигурации

```yaml
ntp:
  enable: true
  write-to-system: true              # Перезаписывать системные часы Keenetic
  server: pool.ntp.org               # Использование глобального стабильного пула
  port: 123
  interval: 30                       # Сверять время каждые полчаса
```

---

### 3. Нюанс совместного использования с KeeneticOS
На роутерах Keenetic уже есть своя встроенная системная служба синхронизации времени. Однако, если в процессе загрузки роутера или сбоя сети системное время Keenetic «уплывает», встроенный модуль NTP в Mihomo выступает в роли железобетонной страховки. Как только Mihomo устанавливает соединение с сетью, он мгновенно корректирует часы роутера, предотвращая бесконечные ошибки шифрования `TLS handshake timeout` на ваших прокси-узлах.




<a id="sec-experimental"></a>

## Экспериментальные функции (Experimental) в Mihomo

Этот блок настраивается внутри секции `experimental:` в конфигурационном файле `config.yaml`. В него выносятся продвинутые, низкоуровневые параметры оптимизации сетевого стека Go (движка `quic-go`), а также специфические методы преобразования IP-адресов, которые могут кардинально повлиять на скорость обработки UDP-пакетов и стабильность QUIC-трафика (HTTP/3) на роутерах.

---

### 1. Подробный разбор параметров (Fields)

*   `quic-go-disable-gso: false` — отключение механизма Generic Segmentation Offload (GSO) для QUIC-трафика.
    *   **Как это работает:** По умолчанию выставлено `false` (GSO включен). Механизм GSO позволяет сетевому стеку ядра формировать и передавать огромные пакеты данных единым куском, перекладывая задачу нарезки пакетов на аппаратный чип сетевой карты. 
    *   **Важно для Keenetic:** Если процессор вашего роутера перегружается при просмотре YouTube (через протокол QUIC/HTTP/3), этот параметр должен оставаться `false`. Но если на кастомной прошивке или старых драйверах Linux начинают без причины рваться голосовые звонки или сыпаться ошибки в лог — принудительное выставление в `true` (отключение GSO) может решить проблему совместимости.

*   `quic-go-disable-ecn: false` — отключение механизма явного уведомления о перегрузках (Explicit Congestion Notification).
    *   **Как это работает:** По умолчанию выставлено `false` (ECN активен). ECN позволяет роутерам и серверам в интернете предупреждать друг друга о перегрузках канала связи, не прибегая к принудительному сбросу пакетов (packet drop). Это снижает задержки (ping) и убирает микрофризы в онлайн-играх.
    *   **Важно для РФ:** Некоторые магистральные ТСПУ провайдеров некорректно обрабатывают или полностью блокируют пакеты с флагами ECN в заголовках. Если вы заметили, что современные QUIC-протоколы (Hysteria2, TUIC) работают нестабильно или скорость скачивания постоянно "скачет" до нуля, выставление `quic-go-disable-ecn: true` поможет стабилизировать соединение.

*   `dialer-ip4p-convert: false` — включение механизма преобразования адресов формата IP4P.
    *   **Как это работает:** IP4P — это специфический метод упаковки классического IPv4-адреса и сетевого порта назначения внутрь одного единого IPv6-адреса. Чаще всего этот протокол применяется в пиринговых p2p-сетях (например, продвинутыми торрент-клиентами или игровыми лобби) для беспрепятственного обхода NAT-ограничений провайдера без использования классического проброса портов (UPnP). Активация (`true`) заставляет Mihomo на лету конвертировать такие адреса для их корректной маршрутизации по правилам.

---

### 2. Практический пример конфигурации

Ниже — ориентир для **типичного роутера / Keenetic**: `dialer-ip4p-convert` оставляем **`false`**, если у вас нет осознанной задачи под p2p/IP4P.

```yaml
experimental:
  quic-go-disable-gso: false         # Оставляем аппаратное ускорение пакетов включенным
  quic-go-disable-ecn: false         # Оставляем умный контроль перегрузок сети
  dialer-ip4p-convert: false         # По умолчанию выкл.; true — только если нужен IP4P (часто p2p/торренты)
```
